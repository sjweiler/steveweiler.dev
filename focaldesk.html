FocalDesk

A custom Wayland desktop environment built to explore what a fast, cohesive, keyboard-friendly Linux desktop can be—without treating the compositor as a monolith.

View source on GitHub Explore the architecture
Role Creator and lead developer

Product direction, systems architecture, Rust development, UI, integration, and testing.

Scope 45-package Rust workspace

Compositor, shell, display manager, shared crates, services, native applications, capture, and AI experiments.

Current state Working alpha

Developed and used by its creator, with changing APIs, experimental subsystems, and expected rough edges.

The challenge

Build a desktop, not just a compositor demo.

A usable Linux desktop has to coordinate rendering, input, windows, workspaces, applications, system controls, capture, and session state. Putting all of that inside one compositor process would make failures harder to isolate and future automation harder to control.

FocalDesk keeps the compositor focused on the display and interaction loop while recoverable services handle application launching, power, notifications, dialogs, controls, settings, and AI requests.

System design

A compositor-centered, service-oriented architecture

Native Wayland and XWayland clients meet at the compositor. Rendering and hardware backends stay close to the frame loop, while typed IPC creates a boundary around supporting desktop services.

01

Keep the frame loop focused

Window management, input, rendering, outputs, and shell behavior remain the compositor's core responsibility.

02

Isolate supporting services

Launch, power, notifications, dialogs, and controls run out of process and communicate through explicit IPC.

03

Gate automated actions

AI requests use visible permission prompts and recorded decisions instead of receiving unrestricted desktop access.

Engineering deep dive

Keep variable-latency work out of the frame loop.

A compositor cannot treat application launches, power operations, notification handling, hardware controls, or model requests like ordinary in-process UI work. Any one of them can block, fail, or wait for a person—while input and rendering still need to remain responsive.

I drew the boundary around what must remain compositor-owned: surfaces, focus, workspaces, input routing, shell state, outputs, and frame coordination. Supporting work moves into recoverable user services and crosses explicit request, response, and event contracts.

Constraint

The desktop must keep drawing when a service is slow.

Long-running work cannot share ownership of the rendering and input loop.

Decision

Use typed IPC and separate user services.

Launch, power, notifications, dialogs, controls, settings, and automation can evolve or restart independently.

Tradeoff

More process boundaries require more discipline.

Message contracts, failure handling, logging, service discovery, and permissions become explicit engineering work.

Result

A smaller critical core with visible safety boundaries.

The architecture isolates recoverable work and gives AI requests a native approval path before an action proceeds.

Working system

Usable prototypes exercise the complete system.

First-party applications are deliberately labeled prototypes. Their job is to test the compositor, service boundaries, visual language, and everyday workflows together rather than in isolation.

Compatibility proof

Tested with real graphics workloads.

A known-working World of Warcraft setup through Wine and DXVK validates more than appearance. It exercises XWayland, Vulkan translation, GPU acceleration, input handling, and high-performance rendering without implying compatibility with every game, driver, or Wine version.
